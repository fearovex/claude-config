Date: 2026-03-19

## ADDED — Supersedes section in proposal.md template

### Requirement: proposal.md always includes Supersedes section

Every proposal.md generated by `sdd-propose` MUST include a `## Supersedes` section that explicitly enumerates what is being removed, replaced, or contradicted.

The Supersedes section MUST:
1. Be present in every proposal (not optional)
2. Clearly state "None — purely additive change" when nothing is superseded
3. List each removed or replaced item with its rationale
4. Distinguish between three types of supersession:
   - **REMOVED**: Feature/behavior/artifact that will be deleted entirely
   - **REPLACED**: Prior implementation that will be swapped out for a new one
   - **CONTRADICTED**: Documented constraint or decision that this change explicitly contradicts

#### Scenario: Pure addition — no removals

- **GIVEN** the change is purely additive (e.g., "add email notification system")
- **WHEN** `sdd-propose` generates the proposal.md
- **THEN** the `## Supersedes` section MUST state: "None — purely additive change"
- **AND** no removal list is needed

#### Scenario: Change removes an existing feature

- **GIVEN** the user requests "remove the periodic membership refresh hook"
- **AND** this hook currently exists in the codebase
- **WHEN** `sdd-propose` captures the intent
- **THEN** the `## Supersedes` section MUST include:
  ```
  ## Supersedes

  ### REMOVED

  - **Periodic membership refresh hook** (src/services/membership-refresh.ts)
    Reason: No longer needed; membership sync is now event-driven
  ```
- **AND** the proposal MUST cross-reference the exploration.md Contradiction Analysis if one was generated

#### Scenario: Change replaces a prior implementation

- **GIVEN** the user requests "implement new auth token refresh instead of the current 4-hour polling"
- **WHEN** `sdd-propose` authores the proposal
- **THEN** the `## Supersedes` section MUST include:
  ```
  ## Supersedes

  ### REPLACED

  - **Periodic token refresh polling** (old: src/auth/token-poller.ts)
    New approach: Event-based refresh with sliding-window expiry
    Rationale: Reduces unnecessary API calls; improves latency-sensitive flows
  ```

#### Scenario: Change contradicts a documented decision

- **GIVEN** the architecture.md states "All user-facing changes MUST preserve backwards compatibility"
- **AND** the proposal requests "remove the deprecated v1 API endpoint"
- **WHEN** `sdd-propose` identifies the contradiction
- **THEN** the `## Supersedes` section MUST include:
  ```
  ## Supersedes

  ### CONTRADICTED

  - **Backwards compatibility guarantee** (documented in architecture.md)
    Contradiction: Removing v1 API breaks legacy client integrations
    Resolution: Assumes v1 is sunsetting; deprecation period MUST be 6+ months
  ```

## ADDED — Conversation context preservation in proposal.md

### Requirement: proposal.md captures explicit user intents from conversation

When the user provides explicit removal or replacement intents in conversation (not just in the change description), `sdd-propose` MUST extract and preserve these intents in a new `## Context` section.

The Context section MUST:
1. Record any explicit statements about what should be removed, replaced, or changed
2. Capture mobile-specific or platform-specific constraints mentioned in conversation
3. Preserve warnings or cautions the user stated (e.g., "be careful with X")
4. Include a timestamp of when the intent was captured

#### Scenario: User provides removal intent in conversation

- **GIVEN** the user states (in conversation) "Remove the in-memory simulation; we need real SharePoint write-back now"
- **WHEN** `sdd-propose` reads the conversation context
- **THEN** the proposal.md `## Context` section MUST record:
  ```
  ## Context (from user conversation)

  ### Explicit Intents

  - **Remove in-memory simulation**: User stated this must be replaced with live SharePoint integration
  - **Reason given**: In-memory was provisional pending integration; now ready to implement real backend

  Captured: 2026-03-19 15:42 UTC
  ```

#### Scenario: User mentions a mobile constraint not in the formal description

- **GIVEN** the user says in conversation "This MUST NOT block mobile — we need sub-500ms response time"
- **AND** the formal change description does not mention mobile or latency
- **WHEN** `sdd-propose` records context
- **THEN** the proposal.md `## Context` section MUST include:
  ```
  ### Platform Constraints

  - **Mobile latency**: Sub-500ms response time required (user stated)
  ```
- **AND** this constraint MUST appear in the proposal Scope section as a non-negotiable requirement

#### Scenario: No conversation context — formal description only

- **GIVEN** the user invokes `/sdd-ff <description>` with no additional conversation
- **WHEN** `sdd-propose` generates the proposal
- **THEN** the `## Context` section MAY be omitted or state: "Context: None — based on formal description only"

## ADDED — Conversion of exploration Contradiction Analysis into proposal action

### Requirement: Contradictions flagged in exploration.md trigger explicit proposal section

If the exploration.md `## Contradiction Analysis` section reports any CERTAIN contradictions, `sdd-propose` MUST create a corresponding section in the proposal.md that acknowledges the contradiction and describes the resolution approach.

#### Scenario: Contradiction is resolved by proposal

- **GIVEN** exploration.md reports: "User requests REMOVAL of hook; contract GUARANTEES hook presence (CRITICAL)"
- **WHEN** `sdd-propose` generates the proposal
- **THEN** the proposal MUST include an explicit section addressing the contradiction:
  ```
  ## Contradiction Resolution

  ### Hook Removal vs. Contract Guarantee

  **Contradiction**: Exploration detected that removing the periodic refresh hook contradicts the documented contract.

  **Resolution in this proposal**:
  - The contract is being SUPERSEDED by this change
  - The contract's 4-hour guarantee is replaced by event-driven refresh (on-demand)
  - The user has explicitly accepted this breaking change
  - Stakeholders who depend on the 4-hour guarantee MUST be notified

  **Risk**: Breaking change; requires coordination with dependent teams
  ```

#### Scenario: No contradictions — no special section

- **GIVEN** exploration.md reports "No contradictions detected"
- **WHEN** `sdd-propose` generates the proposal
- **THEN** no `## Contradiction Resolution` section is needed

---

## Rules

- Supersedes section is ALWAYS present; if nothing is superseded, MUST state "None — purely additive change"
- Contradictions flagged in exploration.md as CERTAIN MUST be acknowledged in the proposal with an explicit resolution statement
- Context section captures user intents from conversation; MAY be omitted if no conversation context exists
- All supersession entries (REMOVED, REPLACED, CONTRADICTED) MUST include rationale
- If the proposal contradicts a spec or architectural decision, the proposal MUST explicitly state the consequence (breaking change, deprecation period, stakeholder coordination)
